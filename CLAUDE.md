# CLAUDE.md — guidance for Claude Code

This is a **player template** for Battle of Code. After fork + signup via `tools/signup.py`, the human owner has a working Rich-UI CLI client and a starter bot. Your job is to help them improve the bot.

## Where things live

- **The bot the user owns**: `bot.py` at repo root. Single file. The only file you edit when the user says "improve my bot".
- **Go / Node stubs**: `bot.go`, `bot.js` at repo root. They print "not implemented yet" and exit 1. Only touch them if the user explicitly asks for a port.
- **Tools** at `tools/`:
  - `keygen.py` — generate Ed25519 keypair (`keys/<u>.key` + `keys/<u>.pub`).
  - `signup.py` — interactive signup: ensure `.env`, generate keys, prompt for the GitHub push, then call REST `/api/auth/signup` + `/api/auth/signup/verify`.
  - `login.py` — WebSocket smoke-test (hello → auth_ok → welcome → ping/pong).
- **Interactive CLI player**: `client/`. Rich UI, WSAD. Same auth as `bot.py` (signed hello). Launch via `./play.sh`.
- **Docs**: `docs/RULES.md` (game), `docs/AUTH.md` (REST signup flow), `docs/API.md` (WebSocket wire format), `docs/EXAMPLES.md` (decision-making cookbook). Translations of the top-level README live in `docs/README_<LANG>.md`.
- **Scripts at root**: `play.sh` (launches `client/client.py`), `validate.sh` (legacy curl check against a future `/api/validate` — keep until that endpoint exists or the script gets retired).

## When the user says "improve my bot"

1. Read `bot.py`. The structure is: imports → `decide(state)` (the one editable function) → plumbing (`run_session`, signing helpers, `main`).
2. Edit `decide(state)` in place. Keep its signature: takes the latest `state` dict (`me`, `view`, `players`, `tick`), returns one of `"W" | "A" | "S" | "D" | "N"`.
3. Read `docs/RULES.md` (skim sections 2 + 3) before strategising.
4. Stateful behaviour (memory across ticks) goes into module-level variables or a small dataclass — don't add classes for the sake of it.

## Wire-protocol invariants

`docs/API.md` (WebSocket gameplay) and `docs/AUTH.md` (REST signup) are the sources of truth. Quick recap:

- **REST auth**: HTTP on `BOC_AUTH_HOST:BOC_AUTH_PORT` (default `:8000`).
  - `POST /api/auth/signup` → server fetches `key.pub` from the player's GitHub fork on branch `<username>`, returns a nonce.
  - `POST /api/auth/signup/verify` → submit hex signature over the nonce bytes.
  - Rate-limited per source IP: 10/h, 50/d.
- **Game**: WebSocket on `BOC_GAME_HOST:BOC_GAME_PORT` (default `:5555`).
  - First frame must be `{"type":"hello","username","ts","sig","is_bot","is_view","follow_rank"}`.
  - `sig` = hex Ed25519 over UTF-8 of `bocbot:hello:<username>:<ts>`. Skew tolerance ±30 s.
  - On success: `auth_ok` + `welcome`, then a stream of `state` (+ event messages: `died`, `captured`, `kill`, `respawn`).
  - Client commands: `{"cmd":"dir","d":"W|A|S|D|N"}`, `{"cmd":"ping","t":...}`, `{"cmd":"quit"}`.
  - 180° turns rejected; 1-tick cooldown after a turn.

There is no token, no JWT, no cookie. Every connect is a fresh signed hello.

## Hard rules

- **Never** commit anything under `keys/`. Git-ignored; don't `git add -A` past it.
- **Never** push `keys/<login>.pub` to `main`. It belongs only on the user's `<github-login>` branch.
- **Don't** add `Co-Authored-By: Claude` trailers, "generated by AI" comments, or robot emoji in code, commits, or PRs. The user's repo, the user's voice.
- **Don't** reintroduce a `bot/` directory or SDK package — `bot.py` is intentionally a single file with no local imports.
- **Don't** add legacy environment-variable fallbacks (`CLIENT_ADDR`, `CODEBOT_SERVER`, etc). The whole repo uses `BOC_AUTH_*`, `BOC_GAME_*`, `USERNAME` only.

## Style

- Python 3.10+, type hints on public functions, snake_case.
- `decide(state)` runs ~10 times/sec. Keep it fast. No file IO, no network calls inside it.
- Logger format already configured: `"%(asctime)s.%(msecs)03d [%(levelname)s]: (%(name)s) %(message)s"`.

## Workflows the user might ask for

| Ask                              | Where you work                                                  |
|----------------------------------|-----------------------------------------------------------------|
| "teach my bot to hunt"           | `bot.py`, expand `decide()` — read `docs/EXAMPLES.md` first     |
| "log every decision to a CSV"    | `bot.py`, open a file once at module scope, write inside `decide()` |
| "add a unit test for decide()"   | new `test_bot.py` at repo root (no framework wired — `pytest`)  |
| "port to Go" / "port to Node"    | replace the stub in `bot.go` / `bot.js`, mirror `bot.py`'s shape |
| "I lost my keys"                 | re-run `python3 tools/keygen.py` + push new pubkey + `signup.py` |

## What this template intentionally lacks

- No CI. Don't add GitHub Actions unless asked.
- No bot manager / multi-bot orchestration. One process, one bot.
- No server-side hosting. Bots run on the player's machine.
- No SDK package. The protocol is inlined into `bot.py` so a newcomer can read the whole client end-to-end in one file.
