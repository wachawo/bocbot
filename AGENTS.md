# AGENTS.md — guidance for AI coding assistants

For Cursor, Aider, Roo Code, Continue, Cline, and similar tools. (Claude Code reads `CLAUDE.md`; this file mirrors its contents.)

## What this repo is

A player template for **Battle of Code**, a multiplayer territory-capture game. The human owner forks the repo, runs `python3 tools/signup.py` to register an Ed25519 key, then either plays manually (`./play.sh`) or edits `bot.py` and runs it.

## Where to edit

- **Bot logic**: `bot.py` at repo root. Single file. The only file you edit when the user says "improve my bot". The editable function is `decide(state)` at the top; the rest is plumbing (signing helpers, reconnect, `main`).
- **Go / Node stubs**: `bot.go`, `bot.js` at repo root — print "not implemented yet" and exit 1. Touch only on explicit port request.
- **Tools**: `tools/keygen.py`, `tools/signup.py`, `tools/login.py` — registration helpers, see CLAUDE.md.
- **Interactive CLI player**: `client/`. Edit only when fixing a UI bug.

## Where to read

- `docs/RULES_EN.md` — game mechanics + anti-patterns (zones, trails, capture, death, `trapped_in_zone`). Russian mirror at `docs/RULES_RU.md`.
- `docs/AUTH_EN.md` — REST signup deep dive (Ed25519 trust model, error codes, key recovery). Russian mirror at `docs/AUTH_RU.md`.
- `docs/API_EN.md` — WebSocket wire format + § 9 with short `state`-consumption snippets. Russian mirror at `docs/API_RU.md`.

## Hard rules

- Never commit `keys/*.key`. Private keys are git-ignored; don't `git add -A` past the rule.
- Never push `keys/<login>.pub` to `main`. It lives on the `<github-login>` branch only.
- No "generated by AI" comments or commit trailers. No 🤖 emoji.
- Don't reintroduce a `bot/` directory or SDK package — `bot.py` is intentionally a single file with no local imports.
- Don't add legacy environment-variable fallbacks (`CLIENT_ADDR`, `CODEBOT_SERVER`, etc.). The repo uses `BOC_AUTH_*`, `BOC_GAME_*`, `USERNAME` only.

## Bot contract

```python
def decide(state) -> str:
    # state matches docs/API_EN.md § 3 `state` message
    return "D"   # one of "W" "A" "S" "D" "N"
```

`decide(state)` runs ~10 times per second. Keep it pure and fast — no IO, no network, no slow imports. Persist memory between ticks via module-level variables or a small dataclass.

## Wire-protocol invariants

`docs/API_EN.md` is the source of truth. Quick recap:

- **REST auth**: HTTP on `BOC_AUTH_HOST:BOC_AUTH_PORT` (default `:8000`).
  - `POST /api/auth/signup` → server fetches `key.pub` from the player's GitHub fork on branch `<username>`, returns a nonce.
  - `POST /api/auth/signup/verify` → submit hex signature over the nonce bytes.
- **Game**: WebSocket on `BOC_GAME_HOST:BOC_GAME_PORT` (default `:5555`).
  - First frame must be `{"type":"hello","username","ts","sig","is_bot","is_view","follow_rank"}`.
  - `sig` = hex Ed25519 over UTF-8 of `bocbot:hello:<username>:<ts>`. Skew tolerance ±30 s.
  - On success: `auth_ok` + `welcome`, then a stream of `state` (+ event messages: `died`, `captured`, `kill`, `respawn`).
  - Client commands: `{"cmd":"dir","d":"W|A|S|D|N"}`, `{"cmd":"ping","t":...}`, `{"cmd":"quit"}`.

No token, no JWT, no cookie. Every connect is a fresh signed hello.

## Style

- Python 3.10+, type hints on public functions, snake_case names.
- Imports: stdlib, then third-party. No `from bocbot import ...` — there is no `bocbot` package.
- No `print()` — use `logging`. Logger format already configured: `"%(asctime)s.%(msecs)03d [%(levelname)s]: (%(name)s) %(message)s"`.
- One concern per file. `bot.py` is intentionally a single file; split only if it grows past ~300 lines.

## What's missing on purpose

- No CI. Don't add GitHub Actions unless asked.
- No tests on the bot (the renderer has its own at `client/test_render.py`). Add `pytest` if the user requests one.
- No bot orchestration. One process, one bot, foreground.
- No SDK package. The protocol is inlined into `bot.py` so a newcomer can read the whole client end-to-end in one file.
